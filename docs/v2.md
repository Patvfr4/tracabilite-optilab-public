# Plan V2 — Architecture & Fonctionnalités

## Vue d'ensemble

La V2 introduit trois grandes évolutions :
1. **Multi-grappe** — plusieurs réseaux de laboratoires dans le même environnement
2. **Système de modules** — fonctionnalités activables/désactivables par grappe ou par labo, configurables entièrement dans l'app
3. **Mode chauffeur PWA** — app mobile offline pour le suivi des bons de départ

---

## Points à clarifier avant implémentation

### 1. Surcharge des modules — grappe peut-elle forcer un module ?

La résolution actuelle est `labo-first` : un labo peut activer ou désactiver un module même si
la grappe a une valeur différente. Est-ce voulu ? Deux modèles possibles :

- **Modèle A — valeur par défaut** : la grappe propose, le labo peut surcharger dans les deux sens.
- **Modèle B — imposition** : la grappe peut forcer un module ON ou OFF, sans surcharge possible par le labo.

Suggestion : ajouter un champ `allow_override: bool` dans `module_config` au niveau grappe pour
distinguer les deux cas. **À décider avant de coder le système de modules.**

### 2. Envois inter-grappes — comptage KPI et visibilité historique

Quand Grappe A envoie un colis vers un labo de Grappe B, des règles métier doivent être définies :

- L'envoi compte-t-il dans le KPI de A, de B, ou des deux ?
- Dans l'historique de Grappe B, l'envoi apparaît-il comme "reçu de Grappe A" ou comme un envoi ordinaire ?
- Comment Grappe B sait-elle qu'un envoi lui a été expédié si les panels sont isolés par grappe ?

**À décider avant de coder la Phase 3 (inter-grappes).**

### 3. Chauffeur multi-grappe — périmètre réel

Le chauffeur est rattaché à un labo (une grappe). Si une Grappe B a ses propres bons de départ
et ses propres chauffeurs, le modèle actuel ne permet pas à un chauffeur de Grappe A de voir
les bons de Grappe B (sauf Admin). Cas à préciser :

- Un chauffeur peut-il être explicitement autorisé à voir plusieurs grappes sans être Admin ?
- Ou la règle est : chauffeur = une grappe, point final ?

**À décider avant de coder la Phase 4b (RLS chauffeur).**

### 4. Sous-labos — définition fonctionnelle

Le module est listé mais pas défini. Deux interprétations très différentes :

- **Alias d'affichage** : un labo a des sous-entités nommées (bio, micro, patho) visibles dans les
  sélecteurs mais sans logique propre.
- **Entité complète** : un sous-labo a ses propres envois, son propre historique, ses propres utilisateurs.

L'interprétation choisie change radicalement la complexité (quelques heures vs plusieurs semaines).
**À clarifier avant d'inclure sous-labos dans le scope V2.**

### 5. Risque RLS — stratégie de test

La matrice de permissions (multi-grappe + multi-labo + admin bypass + chauffeur scope) est complexe.
Un bug RLS ne provoque pas de crash visible — il provoque une fuite de données silencieuse.
Prévoir une phase de test dédiée avec des comptes de chaque rôle dans chaque contexte avant
tout déploiement multi-grappe en production.

### 6. Migration CFG → module_config

La migration des configs existantes (`bonsDepartEnabledLabs`, `hgrappeEnabledLabs`) vers
`module_config` doit être explicite, idempotente et testée séparément. Ne pas l'inclure dans
la migration de schéma — en faire une migration de données distincte avec rollback possible.

| Sujet | Décision |
|---|---|

## Phase 1 — Fondation multi-grappe

### Migration `020_multi_grappe.sql`

```sql
CREATE TABLE grappes (
  id         UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  name       TEXT NOT NULL,
  code       TEXT NOT NULL UNIQUE,  -- ex: 'BSL', 'LAUR'
  active     BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE laboratories ADD COLUMN grappe_id UUID NOT NULL REFERENCES grappes(id);

-- Migrer les données existantes
INSERT INTO grappes (name, code) VALUES ('BSL-Gaspésie', 'BSL') RETURNING id;
UPDATE laboratories SET grappe_id = '<UUID_BSL>';

-- Fonction : grappe de l'utilisateur courant
CREATE OR REPLACE FUNCTION current_grappe_id() RETURNS UUID AS $$
  SELECT l.grappe_id FROM profiles p
  JOIN laboratories l ON l.id = p.labo_id
  WHERE p.id = auth.uid() LIMIT 1;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

### RLS mis à jour

- **Historique / KPI** : filtre `laboratories.grappe_id = current_grappe_id()`
- **Envois reçus inter-grappes** : toujours visibles (`dest_labo_id` appartient à l'utilisateur)
- **Admin** : aucun filtre grappe

### Fichiers impactés

- `envois.js` — filtres grappe dans `loadHistPage`, `loadHistStats`, `loadResumeSentPage`
- `kpi.js` — filtre par grappe courante
- `state.js` — ajouter `currentGrappe`
- `auth.js` — `estAdmin()` bypasse filtres grappe

## Phase 2 — Système de modules

### Migration `021_modules.sql`

```sql
CREATE TABLE module_config (
  id        UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  grappe_id UUID REFERENCES grappes(id),
  labo_id   UUID REFERENCES laboratories(id),
  module    TEXT NOT NULL,
  active    BOOLEAN NOT NULL DEFAULT FALSE,
  config    JSONB DEFAULT '{}',          -- paramètres du module
  CONSTRAINT mc_scope_unique UNIQUE (grappe_id, labo_id, module),
  CONSTRAINT mc_one_scope CHECK (
    (grappe_id IS NOT NULL) != (labo_id IS NOT NULL)
  )
);

-- Résolution : cherche d'abord labo, puis grappe
CREATE OR REPLACE FUNCTION is_module_active(p_module TEXT, p_labo_id UUID)
RETURNS BOOLEAN AS $$
  SELECT COALESCE(
    (SELECT active FROM module_config WHERE labo_id = p_labo_id AND module = p_module LIMIT 1),
    (SELECT mc.active FROM module_config mc
     JOIN laboratories l ON l.grappe_id = mc.grappe_id
     WHERE l.id = p_labo_id AND mc.module = p_module LIMIT 1),
    FALSE);
$$ LANGUAGE sql STABLE SECURITY DEFINER;

CREATE OR REPLACE FUNCTION get_module_config(p_module TEXT, p_labo_id UUID)
RETURNS JSONB AS $$
  SELECT COALESCE(
    (SELECT config FROM module_config WHERE labo_id = p_labo_id AND module = p_module LIMIT 1),
    (SELECT mc.config FROM module_config mc
     JOIN laboratories l ON l.grappe_id = mc.grappe_id
     WHERE l.id = p_labo_id AND mc.module = p_module LIMIT 1),
    '{}');
$$ LANGUAGE sql STABLE SECURITY DEFINER;

-- Migration des configs existantes
INSERT INTO module_config (labo_id, module, active)
  SELECT id, 'bons_depart', TRUE FROM laboratories
  WHERE id = ANY(/* bonsDepartEnabledLabs depuis app_config */);

INSERT INTO module_config (labo_id, module, active)
  SELECT id, 'hors_grappe', TRUE FROM laboratories
  WHERE id = ANY(/* hgrappeEnabledLabs depuis app_config */);

-- Départements : actif + requis par défaut
INSERT INTO module_config (labo_id, module, active, config)
  SELECT id, 'departements', TRUE, '{"required": true}' FROM laboratories;
```

### Module `departements` — requis ou optionnel

Le paramètre `required` dans `config` détermine si le champ Département(s) est obligatoire
dans le formulaire d'envoi. Si `required: false`, le champ s'affiche mais n'est pas bloquant.

```json
{ "required": true }   // Département obligatoire (défaut)
{ "required": false }  // Département optionnel
```

Côté JS dans `saveEnvoi()` :
```js
var deptRequired = window.CFG && window.CFG.modules
  && window.CFG.modules.departements && window.CFG.modules.departements.required !== false;
if (deptRequired && !state.departementsActifs.length) {
  notifier('serr', 'Veuillez sélectionner au moins un département.', 'e'); return;
}
```

### Configuration dans l'app — Panel Config → Modules

Tous les modules sont configurables depuis l'app, sans toucher la base. L'interface :

```
Config → Modules
  ┌─────────────────────────────────────────────┐
  │  Niveau grappe                              │
  │  ● KPI              [ON]                   │
  │  ● Mode Chauffeur   [ON]                   │
  │  ● Intra-grappe     [ON] (toujours actif)  │
  ├─────────────────────────────────────────────┤
  │  Niveau labo — RIMOUSKI                     │
  │  ● Hors-grappe      [ON]                   │
  │  ● Bons de départ   [ON]                   │
  │  ● Départements     [ON]  Requis [ON]      │
  │  ● Sous-labos       [OFF]                  │
  └─────────────────────────────────────────────┘
```

- **Admin** : voit et configure toutes les grappes et tous leurs labos
- **Superviseur grappe** : configure sa grappe et ses labos
- Sauvegarde via RPC `set_module_config(module, scope_id, active, config)`

### Fichiers impactés

- `app-config.js` — charger `module_config` dans `CFG.modules` au démarrage
- `auth.js` — `isModuleActive(module)` côté client (lit `CFG.modules`)
- `index.html` — section Modules dans panel Config
- `public/js/modules.js` — nouveau fichier pour l'UI de gestion des modules
- `envois.js` — `saveEnvoi()` respecte `departements.required`
- `ui.js` — `showPanel()` vérifie `isModuleActive` avant d'autoriser l'accès

Dropdown en haut de la sidebar visible uniquement pour le rôle `admin`. Permet de
basculer le contexte → voir l'historique/KPI d'une autre grappe sans changer de compte.

- Création via SQL documenté (pas d'interface d'onboarding)

- Badge `[Grappe B]` dans les listes pour les envois inter-grappes

## Phase 4 — Mode Chauffeur PWA

### Contexte

Le chauffeur utilise son propre téléphone. À chaque destination, le technicien réceptionnaire
signe sur l'appareil du chauffeur. L'app fonctionne hors-ligne pendant le trajet.

### Décisions

| Sujet | Décision |
|---|---|
| Scope du module | Niveau **grappe** |
| Multi-grappe | Chauffeur peut livrer pour plusieurs grappes |
| Signature | Canvas au doigt, 1 fois par (bon × destination) |
| Qui signe | Technicien réceptionnaire sur le téléphone du chauffeur |
| Vues | Par arrêt + Par bon (toggle) |
| Offline | Cache localStorage au départ, sync auto à la reconnexion |
| Cibles | Safari iOS + Android Chrome |
| Background Sync | NON — logique de queue JS pure |

### Scope grappe

Le chauffeur voit les bons En transit de **toutes les grappes** où le module est actif
et où il est rattaché via son labo. Un chauffeur régulier = une grappe. Un Admin peut
voir toutes les grappes.

### Schéma de données

```sql
-- Table des livraisons : une ligne par (bon × destination)
CREATE TABLE bons_depart_livraisons (
  id              UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  bon_id          UUID NOT NULL REFERENCES bons_depart(id) ON DELETE CASCADE,
  dest_labo_id    UUID REFERENCES laboratories(id),
  dest_ext_lab_id UUID REFERENCES external_labs(id),
  dest_nom        TEXT NOT NULL,          -- dénormalisé pour offline
  statut          TEXT NOT NULL DEFAULT 'en_attente'
                  CHECK (statut IN ('en_attente','livré','problème')),
  date_livraison  TIMESTAMPTZ,
  signature       TEXT,                   -- JPEG base64 compressé
  notes           TEXT,
  tech_nom        TEXT,
  chauffeur_id    UUID REFERENCES profiles(id),
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE bons_depart_envois
  ADD COLUMN livraison_statut TEXT CHECK (livraison_statut IN ('ok','problème')),
  ADD COLUMN livraison_note   TEXT;
```

### Architecture offline

```
Départ (online)
  → Fetch bons En transit de la grappe + livraisons → localStorage
  → JWT frais, app prête

Transit (offline)
  → Lecture cache localStorage
  → Signatures stockées dans ch_queue[]

Arrivée (online ou offline)
  → Online  : sync immédiate via RPC confirm_livraison()
  → Offline : queue → auto-sync sur event 'online'
  → JWT expiré : refreshSession() → fallback "Reconnectez-vous"
```

### Plan de match chauffeur — 4 phases

**Phase 4a — Fondations PWA**
- `public/manifest.json` + icônes
- `public/sw.js` (cache app shell)
- Rôle `chauffeur` via module_config
- `auth.js` : `estChauffeur()`
- `ui.js` : redirections chauffeur dans `showPanel`
- `index.html` : manifest, SW, `panel-chauffeur`, `nav-chauffeur`

**Phase 4b — Données**
- Table `bons_depart_livraisons` + colonnes `bons_depart_envois`
- `create_bon_depart` → auto-crée les livraisons par destination
- `add_envois_to_bon` → crée les livraisons manquantes
- RPC `confirm_livraison(livraison_id, statut, date, signature, tech_nom, colis_problemes[])`
- RPC `mark_arret_probleme(livraison_id, note)`
- RLS chauffeur : scope grappe

**Phase 4c — Interface**
- `public/js/chauffeur.js` (~400 lignes)
- `public/js/signature-pad.js` (~80 lignes)
- Modal signature (canvas + nom + date)
- Modal problème colis (checkboxes + note)
- CSS mobile (grandes zones de tap)

**Phase 4d — Offline + Sync**
- Cache localStorage au chargement
- Queue `ch_queue[]` → localStorage
- Auto-sync sur `online` event
- Badge `[N en attente]` + sync manuel
- Indicateur `[Hors-ligne]`
